World Generation

I'm wiping the slate clean here, I will be using the plugin/mod Terra for all World Generation purposes except for certain structures.

The main reasons for this are customizability, ease of use, compatibility and performance. using datapacks has proven to cause massive lag when it comes to world generation. nothing I do can change that except not using them, they're also harder to modify without breaking something somewhere else and most include their own custom items which cause other headaches as the items don't register properly with plugins.

it's also easier to make sure that the correct world generation is applied to the correct worlds, with datapacks it was a bit hit and miss as they're not properly supported server side with plugins. and then we get to the second reason, I will be adding Multiverse core or a plugin like it to add a lobby world to the server and later a tutorial world.

Multiverse

Multiverse will be responsible for handling all our new Custom worlds making sure we can include test worlds on the server without having to worry that it will destroy the player's progress due to loss of inventory, etc. It will also allow us to include both a Lobby world and a Tutorial world.

Lobby World

The Lobby world will be where the player spawns when they first join the server and they can go visit it agian later by either entering a world portal or using the lobby command which will be implemented. the lobby will have a few fun minigames that can't be played elsewhere on the server and it will have a separate inventory system from the main world.

Tutorial World

The tutorial world will be a lot of work, i'm planning on using it to introduce players to some of the harder-to-understand plugins and as a reference guide for all to go back to if they want to look at which features the server has. those that we can't display easily, will rather have a space available in the tutorial library, where there will be guides that players can use to understand the more complex and vague mechanics included in the server. This will also be where the server's history will be recorded.

Towny Cultures:

  • TownyCultures is an add-on plugin for Towny, which enables "cultures" on Towny servers.
  • Players that share a culture have their own "chat" channel.
  • Towns choose what culture they want to be a part of.

Features:

  • Each town has a 'culture' e.g. "Greek", "Celtic", "Roman", "Azurian" etc.
  • Each town culture is displayed on the Town screen.
  • Town residents can communicate with all other town residents of the same culture (regardless of nation),
    using the Culture Channel: /cc <message>.
  • Town culture is fluid and dynamic. There is no central administrator for each culture.
    A new culture can be created by a mayor, using: /town set culture <culture>.
    The same command can be used to join an existing culture.
  • Nations do not create or specify culture in this way.
    Instead their culture is the sum of whichever cultures their component towns identify with,
    and/or whichever culture(s) they might claim to be affiliated with.
  • Culture is important in war systems where town transfer-after-invasion is possible,
    because although a town might be transferred to an enemy nation,
    the residents can continue communicating with their friends in the same culture,
    using the Culture Channel: /cc <message>.
  • Culture can be shown on your dynmap' town popups.
    Add %culture% in your Dynmap-Towny config's InfoWindow.
  • Culture can be shown using the placeholder: %townycultures_culture%

The Profile System & Website Revamp & Chat Integration

The Profile system is complex and not completely planned out yet, but the idea is to create a unique player ID on the server side to link all your server-related accounts together. that would be your discord, MC, Dynmap, Plan and website account.

It will allow me and the future mod/admin team to handle rule breakers better and it'll also allow for better security when it comes to the accounts linked to the server. I'm looking into adding 2-factor authentication to the website rewrite, but to use that to guard the dynmap and plan on the website as those are also planned to be integrated into the new website.

The website itself will have a profile page where you can see your own profile and it'll include settings to hide your progress from others. when I say progress, I mean it will show your current health as of when you were last online, hunger, defense, inventory, end chest inventory and your backpack inventory. it'll also show all your details that Plan shows, outside of certain details like player ip's and locations. and most of this will also be added to an in-game profile page in the menu system.

I plan on allowing it to also be previewed in Discord chat by a bot command, hence why it's all connected.

Now the website will be greatly expanded upon in the future as I plan on integrating the dynmap and Plan sites into the website while also adding the new profile system, I will be adding forums to the website alongside a wiki that will have all the server documentation on it. the forums I'm still looking into if it is feasible and how many issues it would cause me to integrate the discord with it so that you can read the forums on both the website and here on the server. then I'm also hoping to completely integrate the chat channels between the server, discord, dynmap and the website. but dynmap chat might be removed in favour of just having the chat linked to the website.

Discord Roles

This is a basic version of the role layout it will change heavily in the future and a lot more information will be added once the new rules and punishment systems have been created.

  • Owner - Literally the Owners of the Discord/MC Server and Website. - Will focus mainly on developing and somewhat managing all three
  • Council - The Council members are the heads of they're respective admin teams. there will be 3 admin teams, one for each platform, They handle most of the managerial work of the admin teams and are basically the decision-makers.
  • Admin - Their work is to manage their respective platforms all three platforms admin teams will have the same role on Discord. They will be the ones that look into open tickets and if needed provide support or elevate the issue to the council.
  • Moderator - Same as admin three teams one for each platform handling the grunt work of looking into issues and reporting them to admins.
  • Member - Normal users of the discord server.
